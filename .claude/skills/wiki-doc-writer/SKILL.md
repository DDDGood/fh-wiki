# wiki-doc-writer（攻略主題長文撰寫）

## 核心職責

針對使用者指定的主題（例：「S1 後驅甩尾調校原理」），從 `Docs/_sources/` 全文蒐集相關段落，綜合後產出一份精煉的 `wiki/{category}/{主題}.md`。

**與 wiki-integrator 的差異**：

| | wiki-integrator | wiki-doc-writer |
|--|----------------|-----------------|
| 觸發 | 有新來源進來了 | 想寫某主題長文 |
| 範圍 | 一份來源 vs 全 wiki | 全 _sources vs 一個新主題 |
| 動作傾向 | MERGE / REVISE / DUP 為主 | 幾乎都是 NEW |

---

## 流程

### Phase 1：確認主題與落點

1. 接收主題名稱。
2. 比對 `_category.yml` keywords，建議分類；若使用者有給 `[目標分類]` 參數則以參數為準。
3. 決定檔名（繁中、細粒度）：`wiki/{category}/{主題}.md`
4. **檢查衝突**：Glob 該分類下是否已有高度重疊的檔 → 若有，提示使用者改用 `wiki-integrator` 做 MERGE，或改主題名避開重疊。
5. 告知路徑與預計動作，請使用者確認。

### Phase 2：來源蒐集

1. Glob `Docs/_sources/*.md`，列出所有來源檔。
2. 對每份來源，用 Grep（或 Read 後人工閱讀）找出與主題相關的段落。
3. 產出「**來源盤點**」給使用者看：

```markdown
## 來源盤點：{主題}

| 來源 | 相關章節 | 涵蓋的面向 | 數值重點 |
|------|---------|-----------|---------|
| 硬核調校指南 | §3.2、§4.1 | 原理 + A 級數值 | 差速器 45%/20% |
| QuickTune 指南 | §2.3 | S1 甩尾實戰 | 差速器 60%/0% |
| 萌新攻略第九期 | §5 | 公式化計算 | 差速器 = 馬力/120 |
```

4. 請使用者確認範圍（是否漏掉、是否納入某份不甚相關的）。

### Phase 3：大綱

依據盤點產出大綱給使用者確認，**不要直接下筆長文**。

```markdown
## 大綱：{主題}

1. 前言（為什麼要關心這個主題）
2. 基礎概念
   - ...
3. 實用數值與公式
   - ...
4. 依車種差異（後驅／前驅／四驅）
   - ...
5. 常見誤區
6. 延伸閱讀（相關 wiki 檔）
```

### Phase 4：撰寫

依大綱動筆，遵守下列原則。

#### 寫作原則

1. **不複製貼上來源文字**：用自己的話重述，但關鍵術語保留原文括註。
2. **數值一字不差**：引用任何具體數值時，**必須**附腳註標示來源，例：
   ```markdown
   差速器加速鎖止設為 45%[^1]。
   
   [^1]: 來源：地平線5-硬核調校指南 §3.2
   ```
3. **衝突明示**：來源間數值不同，**不自己挑一個**，用表格併列：
   ```markdown
   | 數值 | 來源 | 適用情境 |
   |------|------|---------|
   | 45% | 硬核調校指南 | A 級公路 |
   | 60% | QuickTune | S1 甩尾 |
   ```
4. **繁中 + 保留專有名詞原拼寫**：BRZ、Evora、Viper、S1、A 組等。
5. **分段短**：避免六七行一整坨，用表格、清單、引用區塊切開。
6. **數值欄位前後端明示**：寫「前 30 / 後 28」不要只寫「30/28」。

#### 檔案模板

```markdown
---
title: {主題}
category: {category}
tags: [...]
related_cars: []
pi_class: []
game: FH5
version: 未標註
status: draft
sources:
  - Docs/_sources/{source1}.md
  - Docs/_sources/{source2}.md
last_updated: {YYYY-MM-DD}
revisions:
  - { date: {YYYY-MM-DD}, note: 初版，綜合 N 份來源 }
---

# {主題}

> 本文綜合 {N} 份來源精煉，數值保留原作，衝突處併列呈現。

## 前言

{為什麼讀者會需要這個主題}

## 基礎概念

...

## {其他章節}

...

## 來源

1. [{來源 1 標題}](../../_sources/{source1}.md) — {相關章節}
2. [{來源 2 標題}](../../_sources/{source2}.md) — {相關章節}

## 相關 wiki

- [{相關檔}](../{category}/{file}.md)
```

### Phase 5：收尾

1. 寫入檔案。
2. 更新 `Docs/wiki/{category}/index.md`（若存在）— 新檔加入清單。
3. 若此分類是首度有 stable 文章 → 建議使用者未來補完時將 `status` 從 `draft` 升為 `stable`。
4. 若專案 wiki 總檔數達到 5，且 `Docs/INDEX.md` 不存在 → 主動建立。

---

## 不可違反

1. **沒有來源就不要寫**：如果 `_sources/` 裡對某主題資訊極少，告訴使用者「資料不足，建議先蒐集更多來源再寫」，而非硬湊。
2. **不自己發明數值**：所有具體數值必須有來源引註。
3. **不複寫既有 wiki 檔**：若目標路徑已存在，退回請使用者改用 `wiki-integrator` 做 MERGE 或換主題。
4. **初稿一律 status: draft**：精煉後由使用者決定何時升級 stable。
5. **每次只寫一份主題**：確保品質。

---

## 失敗模式

| 情境 | 應對 |
|------|------|
| 主題太泛（「調校」） | 請使用者聚焦，例：「調校基礎」「差速器調校」「S1 後驅甩尾調校」 |
| 來源全無相關內容 | 告知「無可用來源」，建議先用 `knowledge-curator` 蒐集再寫 |
| 來源只有一份 | 明示「本篇僅基於單一來源，觀點可能偏頗」並在 frontmatter status 設為 draft |
| 多來源嚴重衝突 | 不硬綜合，保留併列 + 備註各自適用條件 |

---

## 與其他 skill 的關係

- **上游**：`_sources/` 已累積足夠材料
- **平行**：新來源處理由 `wiki-integrator` 負責；本 skill 不處理新來源
- **下游**：產出的 wiki 檔會被 `wiki-site-builder` 收錄發佈
