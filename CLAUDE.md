# Forza Horizon 攻略庫專案

## 專案概述

本專案是 **Forza Horizon 系列（主要為 5 代）的繁體中文攻略庫**。
來源可能是 Bilibili 影片字幕、YouTube、Reddit、官方 Wiki、Steam 指南、PDF、論壇貼文、圖片截圖等，經整理後統一以繁體中文 Markdown 形式收錄於 `Docs/`。

所有對話與輸出**一律使用繁體中文**。

## 文件結構

```
D:\Projects\Forza Horizon\
├── CLAUDE.md                     # 本檔（專案規則與自我改進架構）
├── Docs\
│   ├── _raw\                     # 原始素材暫存區（SRT、PDF、圖片…處理完成後刪除）
│   ├── _sources\                 # 原始整理版（bilibili-to-doc / knowledge-curator 產出，永久保留）
│   │   └── *_整理版.md            # 保留原作者風格與完整內容，不修改
│   └── wiki\                     # ⭐ 精煉後的統一攻略庫（靜態網站發佈內容）
│       ├── _category.yml         # 分類定義（tuning/upgrades/driving/cars/settings/events）
│       ├── index.md              # 首頁
│       ├── tuning\               # 車輛調校
│       ├── upgrades\             # 車輛改裝
│       ├── driving\              # 駕駛操作
│       ├── cars\                 # 車輛介紹（單車或車種類型）
│       ├── settings\             # 遊戲設定
│       └── events\               # 活動與賽事
├── .claude\
│   └── skills\                   # 自訂 Skills
│       ├── bilibili-to-doc\      # SRT 字幕 → _sources/*_整理版.md
│       ├── knowledge-curator\    # 外部來源審查與收錄 → _sources/
│       ├── wiki-integrator\      # _sources/ → wiki/ 去蕪存菁（核心整合）
│       ├── wiki-doc-writer\      # 從 _sources/ 綜合撰寫 wiki 主題長文
│       └── wiki-site-builder\    # VitePress 靜態網站建置與 GitHub Pages 部署
└── memory\                       # 用戶偏好與上下文記憶
```

## 內容流向

```
外部來源（影片／網頁／PDF）
     │
     ├── bilibili-to-doc ──┐
     │                     ├─→ Docs/_sources/*_整理版.md
     └── knowledge-curator ┘         │
                                     ├─→ wiki-integrator ─┐
                                     │                    ├─→ Docs/wiki/**/*.md
                                     └─→ wiki-doc-writer ─┘         │
                                                                    └─→ wiki-site-builder ─→ GitHub Pages
```

**分層原則**：
- `_raw/` = 暫存（處理完刪除）
- `_sources/` = **單一事實來源 A**：原始整理版，保留原作風格，**永不修改**，僅作追溯
- `wiki/` = **單一事實來源 B**：跨來源精煉、去重、統一術語，是對外發佈內容

## Skills 目錄

| Skill | 用途 | 觸發時機 |
|-------|------|---------|
| `bilibili-to-doc` | SRT 字幕 → `_sources/*_整理版.md` | 使用者說「整理字幕／處理 _raw 裡的 SRT」 |
| `knowledge-curator` | 外部來源（網頁／論壇／PDF／影片連結）解析、審查、存入 `_sources/` | 使用者提供 URL／檔案想評估是否納入 |
| `wiki-integrator` | 把 `_sources/` 單份來源拆片段、分類、比對既有 wiki 產生整合計劃（NEW/MERGE/REVISE/DUP）後套用 | 使用者說「整合這份攻略」「精煉／去蕪存菁」「處理 _sources 裡還沒整合的」 |
| `wiki-doc-writer` | 針對指定主題從全 `_sources/` 綜合撰寫 wiki 長文（跨多來源精煉） | 使用者說「寫一篇 XX 的 wiki 攻略」 |
| `wiki-site-builder` | VitePress 網站骨架、側邊欄產生、GitHub Pages 部署 | 使用者說「建網站」「部署」「改側邊欄」 |

## 操作原則

1. **繁體中文優先**：所有輸出、章節名、專有名詞翻譯皆為繁體。簡體來源須全文轉繁並以 Grep 驗證。
2. **數值完整保留**：胎壓、外傾角、阻尼、差速器百分比、車高 cm、齒比等**一個都不能漏**。
3. **車廠／車型／遊戲專有名詞保留原拼寫**：BRZ、Evora、Viper、Dodge、Lotus、S1、A 組、AWD 等。
4. **寧缺勿濫**：若現有攻略已覆蓋相同車種／主題／技巧，就不再重複收錄。由 `knowledge-curator` 做審查判斷。
5. **先讀後寫**：收錄新攻略前先 Glob `Docs/wiki/**/*.md` 列出清單、比對相似度，再決定是否新增或補進既有文件。
6. **刪除要知會**：刪除 `_raw/` 原始素材前先告訴使用者刪哪個檔、輸出在哪，等確認或同一輪訊息一併處理，不要悄悄刪除。
7. **`_sources/` 是聖域**：`bilibili-to-doc` 與 `knowledge-curator` 寫入後，**永不修改**其中檔案。任何精煉、去蕪存菁都在 `wiki/` 做。
8. **wiki 檔必含 frontmatter**：每份 `wiki/**/*.md` 都要有完整 frontmatter（title/category/tags/sources/last_updated/revisions 等），格式見 `wiki-integrator` SKILL.md「Frontmatter 規格」章節。缺一不可，否則靜態網站建置會出錯。
9. **衝突不合併**：多份來源在同一數值上意見不同時，**絕不自己挑一個**，用表格併列 + 各自來源引註，讓讀者自判。

## 自我改進規則

本專案的核心目標是讓 Agent **越用越好用**。以下規則在每次對話中都必須遵守。

### 1. 持續完善攻略庫

- **發現即補充**：處理新來源時若碰到**既有攻略漏掉的車型、組別、版本更新後的新數值**，除了新建文件外，也主動把這些資訊補進最相關的既有 `*_整理版.md`。
- **修正過時資訊**：若發現既有攻略的數值、UI 描述、配件名稱已隨遊戲更新而過時，直接在該檔標註或修正，並在開頭「整理日期」後補上「修訂日期」。
- **統一術語**：若多份攻略對同一概念用不同譯名（例：「差速器」vs「分動器」），選定一個並統一，避免未來讀者混淆。
- **建立索引**：當 `Docs/wiki/**/*.md` 累積到 **5 篇以上**時，主動建立 `Docs/INDEX.md` 依 `_category.yml` 的分類列出所有精煉攻略並附一句話摘要；之後每新增一篇就順手更新索引（靜態網站本身也會自動列出，但 INDEX.md 方便離線查閱）。

### 2. 改進 Skills

- **記錄失敗模式**：若 `bilibili-to-doc` 或 `knowledge-curator` 在執行中遇到新的雜訊模式（例如某 UP 主特有的口癖、某網站特殊的廣告夾層、某 PDF 結構解析失敗），立即把處置方式補進該 skill 的 SKILL.md。
- **擴充繁簡對照表**：`bilibili-to-doc` 的繁簡對照表遇到未收錄的簡體字，處理完就加進去。
- **新增常用操作**：若使用者反覆要求某類操作但現有 skill 無法涵蓋（例如「把多篇攻略合併」「依車型重新分類」），主動設計新 skill。
- **優化流程**：發現 skill 步驟可更精簡、更快、更可靠時，直接改進並在 SKILL.md 記錄原因。

### 3. 建立與改進輔助工具

`scripts/` 下的輔助腳本：

| 腳本 | 用途 |
|------|------|
| `scripts/check_integration.sh` | 掃描 `_sources/` 中尚未被 `wiki/` 整合的原始攻略檔。用法：`bash scripts/check_integration.sh` |

以下情境應主動新增腳本：

- **重複的檢查動作**：例如每次都要 Grep 檢查繁簡殘留、檢查 `-->` 時間軸殘留、掃描重複標題。
- **批次產生索引**：自動掃描 `wiki/**/*.md` 的 frontmatter 產出 `INDEX.md`（目前手動維護）。
- **重複內容偵測**：比對兩份攻略章節標題相似度，輔助 `knowledge-curator`、`wiki-integrator` 的比對 Phase。

工具出了問題或有改進空間，直接修，不要累積技術債。

### 4. 改進 CLAUDE.md 本身

本檔是專案的「總規則」，應隨實際運作情況演進：

- **新增規則**：踩過的坑（例如某類來源常有誤譯、某車型在攻略中有多種譯名）在此留下提醒。
- **更新 Skills 目錄表**：新增或廢止 skill 時同步更新。
- **補充操作原則**：多次對話後發現的新慣例（例如輸出檔名格式、章節骨架偏好）固化成規則。
- **記錄遊戲版本／DLC 範圍**：若攻略開始跨 Forza Horizon 4/5/6，在此標註範圍並說明差異處理方式。

### 5. 學習用戶偏好

- **寫進 memory**：使用者的整理風格偏好（章節深度、是否保留原文括註、是否需要名詞對照表）、常關注的車種組別、常用的來源網站——記錄到 `memory/` 中，讓下次對話直接命中。
- **對待糾正**：使用者糾正譯名、格式、結構時，除了當次修正外，**同時更新 memory 與相關 SKILL.md**，確保不再犯同樣錯誤。
- **調整溝通密度**：偏好簡潔就簡潔；需要完整審查就完整審查。

### 6. 主動維護

- **重複掃描**：收錄新攻略前，除了檢查檔名，也快速掃一遍既有檔的章節標題，避免「檔名不同但內容高度重疊」的隱性重複。
- **_raw 目錄清理**：每次對話開始若 `Docs/_raw/` 有殘留檔，提醒使用者是否該處理或刪除——避免暫存區越堆越多。
- **連結有效性**：收錄時文件頂部「來源 URL」若指向明顯不穩定的來源（個人部落格、可能下架的網盤），在文件中多留一份關鍵資訊摘要，不要只依賴原始連結。
- **版本資訊**：Forza Horizon 5 有週期性 Series 更新可能改動車輛數值，發現攻略明顯對應某個舊 Series 時，在文件開頭註明「整理時對應版本：Series XX」。

### 執行原則

> **改進是靜默的**：不必每次都報告「我更新了某文件」。除非是重大變更（新增／刪除整份攻略、改動 CLAUDE.md 規則、廢止 skill）或需要使用者確認的事，否則順手改完即可。
>
> **改進是漸進的**：不要一次大規模重構所有攻略。在完成使用者主要任務的同時，順手改進接觸到的部分即可。
>
> **改進是謹慎的**：改進 ≠ 亂改。不確定的數值與譯名**寧可留空或括註原文**，也不要寫錯誤資訊誤導讀者。遊戲數值改錯一個 0，對玩家就是一整場比賽的差距。
>
> **改進要可追溯**：對既有攻略做實質修正時（不是純排版），在頂部引用區塊補一行 `> 修訂 {日期}：{一句話說明}`，讓未來的自己／使用者知道動過什麼。
